01 · Pain

NEO hôm nay trả lời như brochure — không như support.

Hành khách nhắn để xử lý vấn đề, NEO trả về điều khoản chung. Khi user nói "tiền đã trừ" — câu trả lời vẫn cùng cấp độ với "hành lý xách tay bao nhiêu kg".

Hành khách Vietnam Airlines gặp lỗi sau thanh toán vé hoặc hành lý mua thêm đang khó biết bước xử lý tiếp theo, vì NEO trả lời theo dạng thông tin chung, hỏi lại chưa cụ thể, chưa phân biệt case rủi ro cao liên quan tiền/vé/dịch vụ, và chưa tận dụng tốt context đã có trong hội thoại. — Pain statement, thin-spec §3
Symptom 01

Trả lời chung chung

Câu hỏi cụ thể về vé / hành lý đều nhận về policy dài, không có bước tiếp theo cụ thể.

Symptom 02

Không phân biệt risk

Case "tiền đã trừ, chưa nhận vé" có rủi ro cao, bot vẫn trả như câu hỏi thường — user dễ tin nhầm.

Symptom 03

Mất context giữa lượt

User đã nói "tiền đã trừ" rồi vẫn bị bot hỏi lại "anh đã thanh toán chưa?".

Symptom 04

Không tách multi-intent

3 vấn đề trong 1 câu → 1 đoạn dài sai trọng tâm, không có thứ tự xử lý.

02 · Evidence (self-use NEO)

Bốn finding cho bốn lỗ hổng.

Mỗi ảnh là một lần dùng NEO thật, một quyết định product cần làm khác đi.

G1 — Multi-intent

User hỏi 3 thứ trong 1 câu, bot trả 1 đoạn dài.

→ SPEC: thêm detectedIssues[] + needsIntentSelection=true, hỏi user chọn xử lý mục nào trước.

G2 — Risk

Case tiền/vé lỗi (rủi ro cao) bị trả như câu hỏi thường.

→ SPEC: deterministic risk guardrails — "tiền đã trừ" luôn riskLevel = High, shouldHandoff = true.

G3 — Missing info

Bot hỏi lại chung chung "anh cần gì".

→ SPEC: per-intent missingInfo[] checklist + concrete nextQuestions[] bằng tiếng Việt.

G4 — Context loss

User cấp context rồi vẫn bị hỏi lại từ đầu.

→ SPEC: knownInfo persist qua các lượt, correction path giữ paymentDeducted=true.

03 · Đã build

Một pipeline. Hai năng lực. Không lệch SPEC.

Triage hẹp (3 intent xử lý sâu) cộng với một lớp realtime mới (tra chuyến bay) — chung một response schema, chung một bộ guardrails.

01

Triage hậu thanh toán

ticket_payment_issue, baggage_addon_payment_issue, unclear_payment_issue. Mỗi intent có riêng missing info, next questions, action checklist, handoff summary.

02

Function calling tool

LLM tự quyết khi nào gọi search_flights. Server proxy tới Aviationstack, normalize sang camelCase, dedupe + sort. Tool result đẩy ngược vào agentic loop, model summarize thành flightResults[] structured.

03

SSE pipeline minh bạch

Mỗi bước (phân loại, tra tool, kiểm rủi ro) gửi stage event ngược về browser. User thấy chính xác bot đang làm gì, không phải dấu chấm bouncing chung chung.

04 · Architecture

Frontend tĩnh ↔ Express ↔ OpenAI (tools) + Aviationstack.

No framework, no build step. Một file server.js giữ secret + guardrails. Frontend chỉ render JSON.

06 · Function calling

Agentic loop — LLM tự quyết khi nào gọi tool.

Không phải LLM cố sinh JSON đẹp. LLM nhìn schema tool, đọc câu user, request tool call. Server execute, trả result vào context. Loop tối đa 3 vòng. Vòng cuối model viết JSON tổng kết.

// server.js — agentic loop
for (let iter = 0; iter < MAX_TOOL_ITERATIONS; iter++) {
  const msg = (await openai.chat.completions.create({
    model: "gpt-4.1-mini",
    tools: [search_flights],
    tool_choice: "auto",
    response_format: { type: "json_object" },
    messages
  })).choices[0].message;
  messages.push(msg);

  if (!msg.tool_calls?.length) {
    return JSON.parse(msg.content); // final JSON
  }

  for (const call of msg.tool_calls) {
    onEvent({ stage: `Đang tra Aviationstack — ${dep} → ${arr}` });
    const result = await callAviationstack(JSON.parse(call.function.arguments));
    messages.push({
      role: "tool", tool_call_id: call.id,
      content: JSON.stringify(result).slice(0, 20000)
    });
  }
}

Vì sao agentic, không 1-shot?

Câu user thường thiếu thông tin (date, airline, route). LLM cần ít nhất 2 turn: turn 1 quyết định gọi tool với args, turn 2 đọc result rồi tổng kết. Lớp guardrails ngoài cùng (Zod parse + risk override) vẫn deterministic.

Server tự xử khi tool fail

  • Free tier limit: tự strip flight_date nếu trùng hôm nay (param này paid-only).
  • Codeshare dedup: server-side dedupe flightResults theo flightNumber sau khi LLM trả.
  • Sort guarantee: server-side sort theo departureScheduled ascending — LLM bị lệch thứ tự cũng được fix.
07 · Risk guardrails

Rule cứng chạy SAU LLM — không tin lời hứa của model.

"Tiền đã trừ" → riskLevel High, shouldHandoff true. Bất kể LLM trả gì.

Trigger High risk

  • "tiền đã trừ" / "bị trừ tiền" / "đã thanh toán"
  • "chưa nhận vé" / "chưa nhận email xác nhận"
  • "dịch vụ chưa xác nhận"
  • paymentDeducted=true từ conversation state cũ (correction path)

Hệ quả tự động

  • riskLevel ép thành High
  • shouldHandoff ép thành true
  • Sinh handoffSummary đủ context cho nhân viên
  • Customer answer phải nói "chưa kết luận giao dịch thành công"

09 · Demo paths

5 đường để test trong 3 phút.

01

Happy — câu hỏi chính sách thường

"Hành lý xách tay được bao nhiêu kg?" → risk Low, NEO hỏi hành trình + hạng vé, không handoff.

02

Low-confidence — mơ hồ tiền/vé/hành lý

"App báo lỗi, tiền bị trừ rồi." → risk High ngay, NEO hỏi vé hay hành lý mua thêm.

03

Failure — failure mode nguy hiểm nhất

"Mua thêm hành lý nhưng app báo lỗi, tiền đã trừ." → High, shouldHandoff, tự sinh handoffSummary có sẵn nút copy.

04

Correction — đổi intent giữa hội thoại

"Không phải vé, tôi đang nói về hành lý mua thêm." → đổi intent sang baggage, vẫn giữ paymentDeducted=true từ turn trước.

05

Flight search — function calling

"Tìm chuyến bay HAN to SGN" → tool fire, 25 chuyến cards đa hãng, sort theo giờ khởi hành, status badge cho từng chuyến.

12 · Sau Day 06

Cái này nên đi tiếp như thế nào.

Tích hợp dữ liệu thật

Booking lookup khi có Vietnam Airlines internal API. Hôm nay prototype không bao giờ xác nhận booking — đó là khoảng trống cần API thật để lấp.

Tool registry mở rộng

Thêm tool airline schedule, route distance, lounge info. Pattern agentic loop sẵn — chỉ cần định nghĩa schema + execute, đẩy vào TOOLS map.

Đo lường

% case High-risk thực sự cần handoff (vs false positive). Latency p95 LLM + tool. Tỉ lệ user tự xử xong vs phải gọi tổng đài. Đo trước khi scale.
